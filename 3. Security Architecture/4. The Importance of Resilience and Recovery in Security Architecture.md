## High availability
-  Redundancy doesn't always mean always available, e.g. redundant systems take time to be powered on/used
- HA means always on, always available, no downtime **at all**
- Implementing multiple appliances costs money
### Server Clustering
-  When 2 servers are combined to operate/appear as a single server, generally configured in the OS
-  Can scale easily by adding/removing some of the servers.
![[Pasted image 20260116230808.png]]
-  Uses a **shared storage** for synchronization
### Load Balancing
-  Load is distributed across many servers using a load balancer, which can:
	-  Balance the load
	-  Add or remove devices to scale/match capacity
	-  Detect Failed systems & removes them
### Load Balancing vs Clustering
In general:
-  Load balancing separates servers, i.e. they don't know of each other's existence, while in Clustering they are integrated together
-  In clustering, integrated devices must have the same OS in general, whereas in Load balancing, they are dependent of each other, with only the load balancer deciding where to redirect data.
## Site considerations
-  Recovery sites must be prepared with synchronized data so when a disaster strikes, business processes failover to the alternate site until the problem is addressed, which may take up to weeks.
- The process of Failover and reverting to the original must be documented.
### Hot Site
-  Exact replica, effectively doubles the cost (double exact hardware, resources, ...)
### Warm Site
-  Mid range between Cold/Hot Site, system partially ready/configured.
### Cold Site
-  Empty building with minimal to no equipment/configuration; cheapest type of recovery sites.
### Geographic Dispersion
-  These sites should be physically different than the main site, to protect against large disasters.
-  Farness introduces logistical challenges.
## Platform Diversity
-  Use multiple platforms (Apps, OSes, Clients, ...) to avoid platform-dependent vulnerabilities.
-  Called "spreading the risk".
-  May increase total number of vulnerabilities  (double platforms <=> double N° of Vulns), but reduces impact upon access (More platforms requires more efforts/exploits from the threat actors).
## Multi-cloud systems
-  Redundancy in cloud providers to protect against cloud outages
-  Can be implemented with **Geographic Dispersion** for data to be both geographically dispersed & cloud dispersed
## Continuity of Operations Planning (COOP)
-  When disasters cause disruptions, there needs to be physical alternatives (**Compensating Security Controls**), e.g. Manual Transactions, Paper Receipts, ...
-  Must be well-documented & tested before a problem occurs.
## Capacity planning
-  Adjust the supply to almost exactly meet the demand; a **balanced** approach.
-  Too much demand causes process disruptions & slowdowns, while Too much supply is money wasted.
### People
-  For services requiring human interaction, e.g. call center, ...
-  Kind of hard to scale amount fast
### Technology
-  Technology must be scalable, and chosen to be like that, since early stages of engineering.
-  Can be strengthened using **load balancing**, **clustering**, **Cloud Services**, ...
### Infrastructure
-  Physical Devices, CPU, Storage, Network, ...
-  Implementing cloud apps will lessen burden on the infrastructure, as they are easy to scale.
## Testing
-  Disaster recovery plan must be tested in a well-defined way to not touch production systems.
-  Disaster recovery plan Generally tests a very specific scenario, then we document and discuss the response.
### Tabletop Exercises
-  Simulating the disaster & the recovery plan is cheaper
-  Implement Analysis lo logistics, with the key people involved
### Fail Over
-  Testing redundant systems if they can maintain High Availability in case of fail.
-  Transition between systems must be transparent to users.
-  A lot of devices have built-in failover functionality, or implement appropriate protocols.
### Simulation
-  Simulating phishing attacks, password requests, data breaches, ...
-  Testing internal security + user awareness against phishing attacks/emails
### Parallel Processing
-  Split a process through multiple parallel CPUs (cores).
-  Allows for improved performance & resiliency.
## Backups
-  Considered a disaster plan, crucial for recovering important data
-  Many different implementations, depends on total amount of data, type of backup, backup media, storage location, backup & recovery software, day of the week, ...
### On-Site vs Off-Site backups
-  **On-Site**:
	-  Same facility as live data
	-  Better performance as no internet is needed
	-  Less expensive.
-  **Off-Site**:
	-  Stored in different location
	-  Data is transferred over Internet/WAN & is available after disaster
	-  Restoration can be performed from anywhere.
-  Companies generally use both, for more copies of data with more restoration options.
### Frequency
-  Depends on data type, backup type & amount of backing sets, ...
### Encryption
-  Backups may contain sensitive information, so we need to encrypt them, especially in cloud context.
-  Recovery keys must be kept safe.
### Snapshot
-  Very relevant in virtual machines & cloud-based environments
-  Instant backup of entire system, with current configuration & data
-  One strategy is taking a snapshot every day, as every snapshot builds upon the previous one.
### Recovery Testing
-  Backups need to be tested to recoverability, and data integrity.
### Replication
-  An ongoing, almost real-time backup, keeps data synchronized in multiple locations
-  So there'll always be a live copy of data to be used in case of disaster
-  Works well with hot sites implementation
### Jouranling
-  Corruption of data if power goes out while writing
-  Journaling is writing data to a storage location, until all writing is finalized, then write that data to its appropriate place
-  If power goes off, journaling data is lost, but at least no corruption on the final location.
## Power Resiliency
-  Electricity is the foundation of what we do, and is generally given to us by 3rd parties.
-  So preparations must be set for both short power outages & long-term outages
### UPS (Uninterruptible Power Supply)
-  Short term backout power for clean & graceful systems shutdown
-  Some of its types:
	-  Offline/Standby UPS: switches to generating power after blackouts.
	-  Line-Interactive UPS: adjusts voltage, best for constant brownouts.
	- On-line/Double-conversion UPS: converts power between main source & battery.
-  Some UPSes contain some additional features, e.g. Auto Shutdown, Battery Capacity, Outlets, Phone line suppression, ...
### Generators
-  Long term power backup based on fuel, can power-up sections/entire building, ...

> <b><u>General strategy:</u></b> Use both, a Generator as a long term power resiliency solution, and a UPS for smooth, no-loss transition from failed main power to the generator
