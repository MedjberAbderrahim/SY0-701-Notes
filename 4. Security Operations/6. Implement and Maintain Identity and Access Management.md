## IAM (Identity & Access Management)
- To ensure flexibility with devices (laptops, mobiles, ...), data locations (cloud, private data center, ...), roles (employee, vendor, ...), granular access/permission control must be implemented.
- The process starts with granting access to user (authentication), and ends with revoking that access from him, it also includes authorization (managing permissions & access), and accounting (tracking resource access)
## Provisioning/De-Provisioning User Accounts
- Includes user account creation, includes details (name, attributes, group permissions, ...) and account removal
- Account creation/removal may happen in case of on-boarding/off-boarding, transfers, promotions, ...
## Permission Assignments and Implications
- Always limits permission to what is enough for their job (no privileged access, ...), generally managed through **group policies**
- A users' own files must stay private to other users
## Identity Proofing
- System must verify who you are upon account issuance to assign appropriate permissions
- Gather information from user (validation), e.g. password, security questions, ...
- Needs verification (passport, in-person meeting), or automated verification
### Identity Proofing vs Authentication
- **Identity proofing** = proving you are a real, specific person before an identity is created
- **Authentication** = proving you are the rightful holder of an already-created identity
## Single Sign-On (SSO)
- Basically 'remember my password' feature
- Allows for repeated access using same credentials without manual re-authentication
- Generally temporary, and must be supported for it to work
### LDAP (Lightweight Directory Access Protocol)
- Protocol for reading/writing [X.500 directories](https://en.wikipedia.org/wiki/X.500) over an IP network, used in Windows Active Directory, Apple OpenDirectory, Novell eDirectory, ...
### SAML (Security Assertions Markup Language)
- Open standard for authentication of users to 3rd party databases

### OAuth (Open authorization)
- Authorization framework with significant industry support (google, ...) which works better with mobile & multi-connection stuff
- Generally used in conjunction with `OpenID Connect` for authentication, as **OAuth is not an authentication framework**
## Federation
- Basically the 'Login with google, facebook, ...' option
- Provide access to other entities (partners, suppliers, ...), SSO, ...
- Organizes authentication & authorization between the 2 organizations, which must establish trust to a certain extent (permissions)
## Interoperability
- Technologies must be compatible with each other
- Usually determined by what is at hand (if OAuth is required, ...)
## Access Controls
- Represents **authorization**, which is managing access permissions to resources through creation & enforcing of policies
- Architecture depends on the the access control model implemented
### Least Privilege
- Give users the least amount of permissions possible in limits of what is needed
- Allows to limits scope of compromise & malicious behavior
### MAC (Mandatory Access Control)
- One of the most strict access control models, assigns a label for each object (confidential, sensitive, ...)
- Administrator assigns permissions of users, i.e. one can read only sensitive data, other can read confidential data, ...
### DAC (Discretionary Access Control)
- Standard & well-known access control model, implemented in most OSes, owner has full permissions to files he create 
- Allows flexibility of permissions, but less security
### RBAC (Role-Based Access Control)
- Depends on pre-defined, reality-mapped roles (i.e. manager, director, ...) with different permissions for each role
### Rule-Based Access Control
- Rules are object-dependent, instead of user-dependent
- System checks ACL if accessing that object is allowed for that user
> Example: Lab network access is only available between 9AM - 5PM
### ABAC (Attribute-Based Access Control)
- A more modern, context-aware access control model, which allows for more complex & flexible management of permissions.
- Grants access based on multiple parameters/criteria, e.g. Resource information, IP address, ...
### Time-of-Day Restrictions
- Can be implemented with other controls
- Very difficult to implement in 24-hour environments, so it can be implemented with time zone
> Example: Training room network inaccessible between midnight & 6AM
## Multifactor Authentication
- Use different methods for better proof of identity, e.g. password, mobile app, GPS location, ...
### Factors
Consists of many factors, e.g. something you know, something you have, ...
- **Something You Know:** One of the most popular, includes passwords, PINs, pattern, ...
- **Something You Have:** More secure, e.g. smart card, USB security key, hardware/software tokens, your own phone (SMS), ...
- **Something You Are:** The most personal factor, e.g. biometric, ... Used in specific situations and not alone
- **Somewhere you are:** Restrict or scrutinize access from unusual IPs, places, GPS, ...
## Password Concepts
### Length & Complexity
- More length & complexity adds entropy and makes it harder to brute force the password
### Age & Expiration
- Passwords should have limited time of use, and must be changed & not re-used after
- Critical systems should have more stricter password expiration policies
### Password Managers
- Should diversity & assign different passwords for different accounts, which makes remembering them impractical
- Stores passwords, password managers themselves must be very secure
- There are enterprise & default built-in password managers in general
- Can have more features, e.g. rating password complexity, automatic secure password creation, ...
### Passwordless Authentication
- Use alternative, passwordless authentication methods to mitigate password problems (re-use, weak by default, ...)
- Can be implemented in conjunction with passwords
## Privileged Access Management Tools
### Just-in-Time Permissions + Password Vaulting + Ephemeral Credentials
- Temporary permission assignment, i.e. grant someone admin for some time
- Account doesn't have those privileges by default, so it's still least privilege
- Primary credentials are stored in a "vault", which decides who gets access to them 
- Give the just-in-time user "ephemeral" credentials that gets deleted later, to protect the real primary credentials
