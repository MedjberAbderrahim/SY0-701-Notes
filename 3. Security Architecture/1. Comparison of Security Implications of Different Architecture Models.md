## Architecture and Infrastructure Concepts
### Cloud
#### Security Responsibility

#### Hybrid Cloud
-  Using multiple cloud providers at the same time increases flexibility but is risky & complex
-  Clouds don't talk to each other, which can open door to incompatibility problems
-  Data is more prone to leaking, with more exposure to the public internet.
#### Third Party Vendors
-  Using third party software/hardware comes with risks & considerations.
-  Include third party in risk assessment procedures, as failure in them implies failure in your services too.
### Infrastructure as Code (IoC)
-  Define your infrastructure as code instead of actual devices, simplifies & makes flexible creating & modifying the infrastructure.
-  Allows for standardization & uniformity of infrastructure.
### Serverless Architecture (Function as a Service - FaaS)
-  The Application is deployed as individual, autonomous, **OS-Independent** functions.
-  Built when needed, saving time & money.
-  similar to SaaS, all OS Security concerns are at the third-party.
### Microservices & APIs
-  Breaking applications into small, independent micro-services, which are to be interacted with using **APIs (Application Programming Interfaces)**.
-  More scalable, resilient than **monolithic architecture**, where the whole app is downloaded.
### Network Infrastructure
#### Physical Isolation
-  To prevent Lateral Movement, certain devices are air-gapped from each other (no physical connection), can be done by target, e.g. web servers in one rack, while databases on another rack.
-  Some Managed Service Providers might do it for high security reasons, separating customers into different, air-gapped switches.
#### Logical Segmentation
-  A more scalable and cheaper solution than assigning networking hardware to each customer is using VLANs.
-  No Inter-VLAN Communication allowed by default, which simplifies network design while accomplishing same results.
#### SDN (Software Defined Networking)
-  The functionality of networking hardware can be split into 3 functional planes of operation.
	-  **Infrastructure Layer / Data Plane:** Process the network frames & packets, i.e. forwarding, trunking, encrypting, NAT, ... 
	-  **Control Layer / Control Plane:** Manages the actions of the data plane, i.e. routing tables, session tables, NAT Tables, ...
	-  **Application Layer / Management Plane:** Configure & Manage the device, i.e. SSH, Browser, API, ...
> <b><u>Note:</u></b> Connection parts are data plane, Routing/NAT tables are control plane, while management ports are management plane.
-  Separating functionality can allow to program that functionality (from hardware to software), which is needed for cloud.
### On-Premises Security
-  While giving more control and granularity, it can be more expensive.
-  Local teams are more expensive, but are completely dedicated to the local infrastructure.
-  Security Changes can take time, as all hardware must be recycled by the company itself.
### Centralized vs Decentralized
-  Most Organizations are decentralized, i.e. many locations, cloud providers, OSs, ...
-  Makes it difficult to manage & secure it all, which begs the need for a centralized security post.
-  Centralized using correlated logs, consolidated log files, comprehensive system status & maintenance.
-  Requires a lot of work, and strong hardware, poses problem of **SPoF (Single Point of Failure)**.

### IoT (Internet of Things)
-  From Sensors, Smart Devices, to Facility Automation.
-  Provide automation & flexibility, but weak security defaults, as their creators aren't security professionals.
### SCADA (Supervisory Control and Data Acquisition) / ICS (Industrial Control System)
-  Includes large machinery in the manufacturing & power generation field.
-  Have a centralized control room (power generation, refining, ...), so all machinery can be managed remotely without the need to go there personally.
-  Requires extensive segmentation, no access from the outside, to avoid critical disasters.
### RTOS (Real-Time Operating System)
-  Linux, Windows, ... are non-deterministic OS.
-  Known for deterministic scheduling, guaranteed execution windows.
-  Required for real time operations (cars, planes, military equipment, ...).
-  Extremely sensitive to security issues, as they need to always be available.
### Embedded Systems
-  A device where software & hardware are all contained as a self-contained and purpose-built device, optimized for only their purpose, e.g. traffic light controllers, medical imaging systems, ...
-  So optimized that they don't/can't include modern security mitigations.
### HA (High Availability)
-  **Redundancy is only a mean to availability**, not availability itself, e.g. redundant devices may need to be powered on manually.
-  HA means **always on,  always available**.
-  Generally includes active/active devices, which means both redundant devices are ON & working **at the same time**.
-  HA Almost always means more costs, need a business decision to know where to stop & take the risk.
## Infrastructure Considerations
### Availability
-  Success is often measured by uptime percentage, such as "five nines" (99.999%).
-  While IT focuses on uptime through redundant systems and monitoring, a security nuance is ensuring resources are **available only to the right people**.
### Resilience
-  Measured by **MTTR (Mean Time to Repair)**, which is the length of time required to replace unavailable components with working ones.
### Cost
-  Calculating value is complex; it includes installation (varies by platform), maintenance (internal or manufacturer), and replacement costs.
-  Accounting factors must be considered: depreciation, **capital expenses (CapEx)**, **operational costs (OpEx)**, and tax implications.
### Responsiveness
-  Refers to the delay between a request and a response; humans are highly sensitive to these delays in interactive applications.
-  Responsiveness varies widely based on the complexity of the specific function or transaction being performed.
### Scalability
-  **Elasticity** is the ability to quickly expand or contract an application's footprint based on usage (e.g., end-of-quarter spikes).
-  Security tools must be capable of monitoring the entire application regardless of how it scales.
### Ease of Deployment
-  Cloud deployments often use **Orchestration**, which is the automated, on-demand building of cloud infrastructure.
-  Effective project planning is vital to ensure all resources (hardware, budget, personnel) are ready to avoid implementation delays.
### Risk Transference
-  Organisations may transfer risk to third parties via **Cybersecurity Insurance**.
-  This can provide financial recovery for ransomware losses, outages, business downtime, and legal proceedings.
### Ease of Recovery
-  Efficiency is key; for example, recovering from an image backup might take 10 minutes, whereas a fresh OS installation could take an hour.
-  Recovery processes should be planned during the implementation of new services to find the least expensive path back to a working state.
### Patch Availability
-  Patching is a constant process to fix bugs, provide security updates, and improve availability.
-  Standard IT procedure involves testing patches in a non-production environment before deploying them to avoid breaking working systems.
### Inability to Patch
-  Often found in **embedded systems** (like HVAC controls or time clocks) that lack connectivity or manufacturer support.
-  If a system cannot be patched, it remains vulnerable; security must be provided through other layers, such as firewalls.
### Power
-  Critical infrastructure that requires monitoring and planning from licensed professionals.
-  Redundancy is provided through **UPS (Uninterruptible Power Supply)** for short-term needs or **generators** for total power requirements.
### Compute
-  The "heavy lifting" and processing part of an application, often referred to as a **Compute Engine** in cloud environments.
-  These resources can be scaled to meet specific application requirements across local or cloud-based processors.
