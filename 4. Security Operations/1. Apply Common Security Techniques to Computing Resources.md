## Secure Baselines
-  The security of an application environment should be well-defined and constantly updated (firewall settings, patch levels, ...)
-  Should always be checked, i.e. for integrity, against well-documented baselines, ...
### Establish Baselines
Baselines must be created, and they can be based on already established baselines of:
-  Application Developer
-  OS Manufacturer (e.g. [Security Baselines of Windows](https://learn.microsoft.com/en-us/windows/security/operating-system-security/device-management/windows-security-configuration-framework/windows-security-baselines))
-  Appliance Manufacturer
### Deploy Baselines
After baselines are created, they must be deployed to the actual infrastructure, this can be in 2 forms:
-  Centralized: usually though an administrative console
-  Decentralized: through multiple deployment mechanisms, e.g. Active Directory, MDM (**Mobile Device Management**), ... 
Deployment of large baselines must be automated.
### Maintaining Baselines
-  Most Baselines are best practices & rarely changes, while others need some updating depending on new vulnerabilities, updates, ...
-  Test baselines of known sources to get the best custom baseline for current scenario.
## Hardening Targets
-  Systems by default aren't hardened enough, so we look for **hardening guides**, from 2 sources mainly:
	-  Official: straight from the Manufacturer
	-  Unofficial: in case the manufacturer didn't provide one, generally from community/3rd party.
### Mobile Devices
-  Mobile Devices are a prime target for exploitation, we harden them with:
	-  Updates: bugfixes & vulnerabilities patches
	-  Segmentation: Separation of company & user data.
-  For large scale mobile protection, we can use an MDM (Mobile Device Manager)
### Workstations
-  User desktops & laptops of varius OSes (Linux, Windows, MacOS, ...) must always be monitored, updated & cleaned from unnecessary software.
-  Pacthing & updated can (& should) be automated
### Network Infrastructure Devices
Purpose-Built OSes with limited access, can be hardened by:
-  Implementing Authentication & don't use default credentials.
-  Patch devices, as updates from manufacturer are very important due to their rarity.
### Cloud Infrastructure
-  Secure the cloud management workstation using **Least Privilege** & **EDR (Endpoint Detection & Response)**
-  Backups alternative cloud providers is a best practice.
### Servers
-  Constant updates.
-  Account limitations and setup of a **minimum password length** policy.
-  Always monitored & secured using anti-viruses & anti-malwares.
-  Limit network access.
### SCADA / ICS
-  Generally controlled from a centralized control system.
-  Must be isolated from the internet, generally by an air-gap.
### Embedded Systems
-  Patch devices, as updates are very important due to their rarity.
-  Must be put in a segmented network, and additionally secured using a firewall.
### RTOS
-  Must be isolated from other areas, and ran with minimum services
-  If networking is needed, Can be further protected by a separate firewall or a host-based security technology.
### IoT
-  Patch devices, as updates are very important due to their rarity.
-  Generally insecure by design, always secure & change default credentials
-  Segment them by putting 'em on their own VLANs

## Mobile Solutions
### MDM (Mobile Device Management)
-  Central management for both Company-Owned & user-owned devices
-  Can be used to set security policies, permissions, segmentation, ...
### Deployment Models
#### BYOD (Bring Your Own Device)
-  when an employee brings their phone to work to be used both for personal use & work
-  Must be modern enough to support being managed by the company's **MDM (Mobile Device Management)**
-  Must be configured with appropriate security policies, and have certain policies when selling/trading the device (delete all data, ...)
#### COPE (Corporate-Owned, Personally Enabled)
-  Company-Owned device bought for employees for both personal & work purposes
-  Similar security policies to **BYOD (Bring Your Own Device)**, similar concept to Company-Owned Laptops & Desktops.
#### CYOD (Choose Your Own Device)
-  Same as **COPE (Corporate-Owned, Personally Enabled)**, but the user chooses the device.
### Connection Methods
#### Cellular
-  Generally 4G or 5G
-  Separates geographical areas into Cells, w antennas converge a cell with certain frequencies.
-  Not that secure, prone to traffic monitoring, Location tracking, global remote access, ...
#### Wi-Fi
-  Similar concerns to other Wi-Fi devices, especially in public Wi-Fis (shop, airport, ...)
-  Prone to:
	-  Data capture: use encryption w VPN
	-  On-path attacks (Man-In-The-Middle), and a DoS attack through frequency interference.
#### Bluetooth
-  High speed communication over short distances (PAN), which connects phones, headsets, ...
-  Do not connect to unknown devices
## Wireless security settings
Wireless networks are :
-  Easy to sniff, must implement encryption.
-  Easy to access, Implement proper authentication.
-  Prone to interference or signal problems, implement **MIC (Message Integrity Check)**.
### WPA3 (Wi-Fi Protected Access 3)
-  WPA2 is crack-able by cracking the hash from the 4 way handshake, so WPA3 was introduced in 2018
-  Uses **GCMP (Galois/Counter Mode Protocol)** Block Cipher Mode stronger encryption than WPA2
-  GCMP Also offers data confidentiality using **AES (Advanced Encryption Standard)**, and integrity via **GMAC (Galois Message Authentication Code)**.
-  WPA3 beats WPA2 weaknesses by implementing a mutual authentication with a shared session key, using **SAE (Simultaneous Authentication of Equals)**
### Wireless Authentication Methods
-  We use centralized authentication (802.1X) to replace the not secure enough **PSK (Pre-Shared Key)**
-  Allows for better authentication (username + password), and manage separate credentials.
-  Can be configured in the wireless router's settings, some modes:
	-  WPA3-Personal / WPA3-PSK: Pre-Shared key, usually available in homes, same for WPA2
	-  WPA3-Entreprise / WPA3-802.1X: Authenticates users separately using a centralized authentication server, e.g. RADIUS, generally used in enterprises.
### AAA
Steps:
-  Identification: who you claim to be (usually your username)
-  Authentication: Prove that you really are who you claim to be (password of the username, 2FA, ...)
-  Authorization: Based on identification & authentication, what access do you have ?
-  Accounting: Records resources used & action, e.g. login time, data sent, ...

### RADIUS (Remote Authentication Dial-In User Service)
-  One of the most common & widely available AAA Protocols
-  Centralizes authentication for users to routers, switches, Servers, Remote VPN, ...
## Application Security
-  Best security practices must always be applied and not be neglected in favor of speed or performance
-  Apps must be tested & assured using quality assurance.
### Input Validation
-  Document and check for all input methods, e.g. forms, fields, types, ...
-  Input Validation checking can be thoroughly verified using fuzzers
### Secure Cookies
-  Avoid storing sensitive information within cookies
-  Send cookies only on secure connections, i.e. HTTPS
### Static Code Analyzers
-  Static Application Security Testing (SAST)
-  Analyzes application's code to identify generic security vulnerabilities, e.g. buffer overflows, database injections, ...
-  Very helpful to detect vulnerabilities, doesn't detect everything though (implementation bugs, ...)
-  Need to verify outputs, as it may contain false positives.
### Code Signing
-  Applications signed by a trusted CA (or the developer) to ensure integrity
## Sandboxing
-  To ensure the app only has the appropriate degree of access & authorization
-  Already used in many contexts, e.g. VMs, Mobile Devices, Browser iframes, ...
## Monitoring
-  Generally built-in into the application by the developers
-  Generally used to detect anomalies & audit/analyze them later
